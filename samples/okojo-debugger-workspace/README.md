# Okojo Debugger Sample Workspace

This folder is a ready-to-run Okojo debugger sample with real TypeScript source maps.

## Open

## Run without VS Code

From the repository root:

```powershell
powershell -ExecutionPolicy Bypass -File .\samples\okojo-debugger-workspace\start-debug-server.ps1
```

Or start with an original-source breakpoint immediately:

```powershell
powershell -ExecutionPolicy Bypass -File .\samples\okojo-debugger-workspace\start-debug-server.ps1 -Break src\entry.mts:10
```

This launches `src/Okojo.DebugServer/Okojo.DebugServer.csproj` with `--enable-source-maps`, so `break`, `stopped`, and stack locations use the `.mts` files instead of the generated `dist/*.mjs` files.

## Open in VS Code

Open `samples/okojo-debugger-workspace` in VS Code, then run:

- `Okojo: Sample Launch (generated JS)` for generated `dist/*.mjs`
- `Okojo: Sample Launch (TS source maps)` for authored `src/*.mts`

## What it shows

- launch-time stop on entry
- module entry with multiple files and `import`
- real `.mts -> .mjs + .mjs.map` output generated by TypeScript
- `debugger;` pause points
- generated-JS debugging without source-map remapping
- source-map debugging with breakpoints on `src/*.mts` files
- console output that shows module load order and runtime flow
- stack frames with source paths
- locals and captured scope values from the pause snapshot

## Notes

- The generated runtime entry is `dist/entry.mjs`, but the authored sources live under `src/`.
- Rebuild the generated `.mjs` and `.mjs.map` files with `npx -p typescript tsc -p .\samples\okojo-debugger-workspace\tsconfig.json`.
- The launch config points back to `src/Okojo.DebugServer/Okojo.DebugServer.csproj`.
- For generated JS debugging, open `dist\entry.mjs` or `dist\lib\math.mjs` and use `Okojo: Sample Launch (generated JS)`.
- For TypeScript debugging, open `src\entry.mts` or `src\lib\math.mts` and use `Okojo: Sample Launch (TS source maps)`.
- The sample is intentionally small so you can step through it and verify the debugger UX quickly.
- `checkInterval` is set to `1` in the sample so stepping works immediately while you try the debugger UX.
